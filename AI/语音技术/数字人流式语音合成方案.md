# 数字人流式语音合成 (LLM-Stream-TTS) 方案

在数字人交互中，流畅度取决于 **TTFA (Time to First Audio)** 和 **音频连贯性**。

## 1. 总体架构
```text
[LLM Stream Out] --> [Text Aggregator] --> [Sentence Queue] --> [TTS Engine] --> [Audio/Lip-Sync Queue]
```

## 2. 关键工程实践

### 2.1 文本聚合与切分 (Text Splitting)
- **正则表达式**: `[^。！？！？
]+[。！？！？
]+`
- **逻辑**: 
    1. 接收 LLM Token。
    2. 存入 Buffer。
    3. 匹配到标点符号时，提取该句并推送至队列。
    4. 针对首句：遇到第一个逗号即切分，以牺牲极小韵律感换取快速响应。

### 2.2 并行管道 (Parallel Pipeline)
- **并发处理**: 必须使用 `asyncio` (Python) 或多线程，确保文本生成、语音合成、音频播放三个环节互不阻塞。
- **音频拼接**: 在前端使用 Web Audio API 或播放器队列，确保两段音频之间无缝切换，避免“咔哒”声。

### 2.3 GPT-SoVITS 专项优化
- **API 模式**: 采用 `v2` 版本的推理接口。
- **参数调优**:
    - `top_k`, `top_p`, `temperature`: 适当调低可加快 GPT 部分的采样速度。
    - `batch_size`: 设为 1（实时场景不需要批处理）。
- **后端加速**: 推荐使用 [GPT-SoVITS-Inference](https://github.com/RVC-Boss/GPT-SoVITS) 的 TensorRT 分支。

## 3. 性能瓶颈分析
| 环节 | 典型耗时 | 优化目标 |
| :--- | :--- | :--- |
| LLM 首字 | 500ms - 2s | 使用流式输出，选择快速模型 (如 DeepSeek, GPT-4o-mini) |
| 文本切分 | < 1ms | - |
| TTS 合成 (首句) | 300ms - 800ms | TensorRT 加速, 减少首句长度 |
| 网络传输 | 50ms - 200ms | 使用 WebSocket 或本地部署 |

## 4. 推荐工具链
- **文本切分**: `langchain-text-splitters` 或自定义正则。
- **音频流处理**: `pyaudio` (Python) 或 `Web Audio API` (前端)。
- **同步驱动**: `Wav2Lip` 或 `Live2D` 监听音频流频率数据。
