# 网页端交互数字人助手实现方案

本文总结了 2026 年主流的网页端交互数字人技术架构，涵盖从 2D 看板娘到 3D 写实数字人的全链路实现。

## 1. 核心架构 (The 4-Step Loop)

一个完整的交互循环包括：
1. **ASR (感知)**：Web Audio API 采集 -> Whisper (WASM/API) 转文本。
2. **LLM (大脑)**：DeepSeek/GPT-4o 处理流式文本。
3. **TTS (表达-声)**：Edge-TTS/GPT-SoVITS 生成流式音频。
4. **Drive (表达-视)**：将音频实时映射为视觉动作。

## 2. 视觉驱动分级方案

### 方案 A：轻量级 2D (Live2D)
- **技术栈**：Live2D SDK for Web, PixiJS.
- **同步原理**：通过 `AnalyserNode` 获取音频频率，直接操作模型参数 `ParamMouthOpenY`。
- **优点**：性能极佳，适合移动端和低配设备。

### 方案 B：写实 3D (Three.js)
- **技术栈**：Three.js, Ready Player Me (RPM), Viseme.
- **同步原理**：利用 TTS 返回的 Viseme (位素) 数据，驱动模型的 `morphTargetInfluences`。
- **参考**：[[数字人流式语音合成方案]]。

### 方案 C：高保真视频流 (WebRTC)
- **技术栈**：WebRTC, MuseTalk, FastAPI.
- **同步原理**：后端 GPU 实时推理生成视频帧，通过 WebRTC 轨道传输。
- **优点**：真人级画质，无视觉瑕疵。

## 3. 实时性优化最佳实践 (2026)
- **全链路流式化**：从 LLM 的第一个 Token 开始，立即触发 TTS 和动画预加载。
- **VAD (静音检测)**：在前端集成 VAD，自动判断用户何时结束说话，减少无效请求。
- **WebAssembly**：将 VAD 和小型 ASR 模型部署在浏览器端，实现毫秒级响应。

## 4. 推荐开源框架
- **LiveTalking (LiveTalk)**：2026 年最火的实时流式框架。它通过自校正蒸馏技术将 LivePortrait 和 MuseTalk 结合，实现了亚秒级的亚像素驱动，支持 WebRTC/RTMP 流式输出。
- **Fay**：全能型数字人框架，支持多种驱动及丰富的业务逻辑。
- **Linly-Talker**：集成多种视觉生成算法，适合实验对比不同模型效果。
- **OpenClaw**：参见 [[端侧AI助手-OpenClaw]]。

## 5. LiveTalking vs Fay 深度对比

| 特性 | Fay | LiveTalking |
| :--- | :--- | :--- |
| **定位** | 业务逻辑集成框架 (多功能、重业务) | 实时渲染驱动框架 (高性能、重技术) |
| **优势** | 模块化程度高，内置记忆与带货逻辑 | 延迟极低 (亚秒级)，音画同步效果好 |
| **视觉驱动** | 支持广泛 (UE5, Live2D, Unity) | 专注 2D/3D 生成 (MuseTalk, ER-NeRF) |
| **适用场景** | 虚拟客服、AI 直播间、企业形象 | 实时视频通话、高频交互助手 |
| **硬件要求** | 业务端低，但 3D 渲染端要求极高 | 依赖高性能 GPU (如 RTX 4090/T4) 进行 AI 推理 |

## 6. 行业应用案例：企业展厅数字人

**场景描述**：访客通过语音唤醒大屏上的数字人，咨询公司业务或产品信息。

**方案推荐：Fay + UE5 MetaHuman**
- **推荐理由**：
    1. **视觉档次**：MetaHuman 的影视级渲染效果更符合企业展厅的科技感定位。
    2. **唤醒与集成**：Fay 内置了完整的语音唤醒词处理和 VAD 逻辑。
    3. **业务能力**：方便接入定制的 RAG（本地知识库）系统，确保回答的准确性。
- **实施建议**：
    - 使用 RTX 4090 显卡实现 UE5 本地渲染。
    - 结合 [[数字人流式语音合成方案]] 降低 TTS 导致的等待感。
