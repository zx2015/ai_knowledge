# System Prompt 在 SFT 中的深度影响

在微调大模型时，是否在 QA 对中包含 **System Prompt（系统提示词）** 并非一个简单的格式选择，而是一个涉及模型神经网络激活与概率分布对齐的工程决策。

---

## 1. 人格锚定效应（Persona Anchoring）

System Prompt 在训练中起到了“激活特定权重子集”的作用。

### 1.1 概率空间的聚焦
LLM 是一个概率预测器。当输入中包含特定的 System Prompt（如“你是一个量子物理学家”）时，模型在 Attention 机制中会将权重注意力引导至与其相关的专业词汇空间。
- **训练价值**：在 SFT 时加入 System Prompt，可以帮助模型在更新参数时，将新知识（Answer）与特定的上下文（System）建立强耦合，从而提高专业回答的精准度。

---

## 2. 消除分布偏移（Inference Consistency）

这是最有必要加入 System Prompt 的理由：**保持训练与推理的一致性。**

### 2.1 训练-服务偏差（Training-Serving Skew）
现代模型（如 Qwen2.5, Llama-3）在推理时几乎 100% 会带有一个 System 角色（通常默认是 "You are a helpful assistant"）。
- **现象**：如果你微调时只喂 `User: xxx, Assistant: yyy`。
- **后果**：推理时一旦加上 System Prompt，整个输入的 Token 序列分布会发生改变。对于模型来说，这就像是换了一个人跟它说话，原本在微调中学到的“肌肉记忆”可能无法被触发，导致准确率骤降 20%-40%。

---

## 3. 提升指令遵循（Instruction Following）的鲁棒性

通过在训练集中包含多样化的 System Prompt，可以让模型学会区分“全局指令”与“单次任务”。

### 3.1 权重竞争与优先级
- 如果没有 System Prompt，模型可能会混淆“任务目标”和“表达风格”。
- 加入 System Prompt 后，模型在微调过程中能学会一种层级化的理解方式：System 层负责“我是谁/我该怎么说”，User 层负责“我要做什么”。

### 3.2 多样化训练与鲁棒性 (Robustness Training)
为了防止模型对特定的 System Prompt 产生“字面依赖”，工业界的进阶做法是：
- **同义改写**：对同一个任务，在训练集中使用 3-5 种表达方式略有不同的 System Prompt（例如：“你是一个助手”、“请作为助手帮我”）。
- **空 Prompt 占位**：在 10%-20% 的训练数据中留空 System Prompt，训练模型在没有任何显式背景的情况下，也能根据 User 指令准确判断任务。
- **作用**：显著提升模型在推理时的灵活性，使其对提示词的微小变动不敏感。

---

## 4. 关键风险：未 Mask 导致的容量浪费

这是新手最容易犯的错误：**在计算 Loss 时包含了 System Prompt 部分。**

- **物理后果**：模型会分出宝贵的参数空间（特别是 Rank 值较小时）去学习如何完美地预测出那句“你是一个资深的程序员”。
- **结论**：System Prompt 必须作为**背景（Context）**存在，而不是作为**学习目标（Target）**。在 LLaMA-Factory 中应通过 `train_on_prompt=False` 确保对其进行损失掩码（Loss Masking）。

---

## 5. 总结：到底要不要给？

**结论：强烈建议给出。**

### 5.1 黄金规则
1.  **如果你的任务是通用的**：使用 3-5 种不同表述的默认助手提示词。
2.  **如果你的任务是专业的**：固定使用该领域的专家提示词。
3.  **永远确保**：训练时用的 System Prompt 格式，与你未来推理（部署）时用的格式 **百分之百字符对齐**。
