# 微调语料自动化生成与构造全量方案

在 SFT（有监督微调）阶段，数据的“质”远比“量”重要。一份生产级的微调语料不仅需要精准的指令，更需要科学的结构化设计与多样性增强。本指南旨在构建一套从数据采集、结构化构造到自动化增强的完整闭环。

---

## 1. 核心数据模型：PIA 结构

为了确保模型能够深刻理解任务背景并保持人设稳定，建议全量采用 **PIA (Persona - Instruction - Answer)** 结构。

### 1.1 组件拆解
- **P (Persona / System Prompt)**：定义模型的全局约束。
    - *示例*：“你是一个具备 20 年经验的 Linux 内核工程师，回答时请保持客观、底层，并提供具体的 C 语言代码示例。”
- **I (Instruction / Question)**：具体的任务描述或用户提问。
    - *示例*：“如何分析内核死锁（Deadlock）？”
- **A (Answer / Completion)**：标准答案，包含完整的推理轨迹（CoT）。
    - *示例*：“分析内核死锁首先需要检查 `dmesg` 中的 `lockdep` 日志... [详细步骤] ...”

### 1.2 物理存储格式 (JSONL)
```json
{
  "system": "Persona strings",
  "instruction": "Task description",
  "input": "Additional context if needed",
  "output": "Golden answer",
  "history": [["prev_q", "prev_a"]] 
}
```

---

## 2. System Prompt 的深度构造策略

### 2.1 角色注入（Role Injection）
不仅仅是“你是一个助手”，而是通过 System Prompt 注入**专业知识密度**。
- **弱注入**：“你是一个翻译。”
- **强注入**：“你是一个精通 18 世纪文学风格的法中翻译家，请使用半文言文进行翻译。”

### 2.2 负样本约束（Constraint Gating）
在 System Prompt 中加入“禁令”，并在 Answer 中体现对禁令的遵守。
- *例*：“禁止提及任何外部竞争产品。” -> 在回答中如果涉及对比，模型应自主避开竞品名称。

---

## 3. 自动化语料增强（Data Augmentation）

对于 7000 条这种中等规模的数据，通过自动化手段将其扩展至 2 万条并提升质量是核心工作。

### 3.1 指令改写 (Instruction Paraphrasing)
- **技术手段**：使用 GPT-5.5 对原有的 7000 条指令进行语义等价的改写。
- **目的**：提升模型对不同问法（如：“怎么做”、“求教”、“给我写个”）的鲁棒性。

### 3.2 自主进化 (Self-Instruct / Evol-Instruct)
- **原理**：基于现有样本，让大模型生成“更难”、“条件更多”的衍生问题。
- **效果**：迫使模型在微调中学习处理更复杂的边界情况，直接突破 75% 的逻辑瓶颈。

### 3.3 合成对话链（Synthetic Conversation Chain）
- 模拟多轮对话，将单轮 QA 拆解为多轮引导，培养模型的上下文记忆能力。

---

## 4. 关键工程细节：Loss Masking 与权重分配

### 4.1 为什么要对 System/Instruction 进行 Mask？
在 SFT 训练中，模型只需学习**生成答案**。
- **Mask 原则**：计算交叉熵损失时，将 `system` 和 `instruction` 对应的 Token 权重设为 0。
- **风险**：如果不 Mask，模型会把 Rank 容量浪费在背诵你的 System Prompt 上，导致真正的逻辑拟合能力（r=8 时的核心瓶颈）被稀释。

---

## 5. 2026 质量控制准则 (Data Cleaning)

针对准确率低下的诊断，必须执行以下数据清洗动作：
1.  **逻辑自洽性检查**：使用模型自动对数据集进行打分，剔除 Answer 逻辑跑通、事实错误的样本。
2.  **多样性去重**：使用 Embedding 聚类，删除语义重复率超过 95% 的样本，防止模型在局部数据上过拟合。
3.  **格式对齐**：确保所有控制符（如 `\n`, `###`, `<|im_end|>`）在数据集中与推理模板字面一致。
